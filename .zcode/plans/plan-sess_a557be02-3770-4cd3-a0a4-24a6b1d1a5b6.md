# خطة: تقسيم ملفات المشاركة إلى محطات عمل منفصلة حسب المسؤول

## الهدف
بدل صفحة واحدة شاملة بـ8 تبويبات (يضيع الموظف فيها)، نبني **محطات عمل** صفحات مستقلة — كل موظف يدخل محطته ويرى شغله فقط. البيانات تبقى موحدة في `registrations` (لا تكرار)، نفصل الواجهات فقط.

## البنية
```
/dashboard/participation-cases              ← قائمة كل الملفات (admin) + إنشاء جديد
/dashboard/participation-cases/work/intake   ← محطة الاستقبال (إنشاء + بحث عميل)
/dashboard/participation-cases/work/clients  ← محطة بيانات العملاء والجوازات
/dashboard/participation-cases/work/payment  ← محطة المالية والدفع
/dashboard/participation-cases/work/registration ← محطة تسجيل الفعالية والـBadge
/dashboard/participation-cases/work/invitations   ← محطة الدعوات
/dashboard/participation-cases/work/visa     ← محطة الفيزا والتأمين
/dashboard/participation-cases/work/qc       ← محطة التدقيق والجودة
/dashboard/participation-cases/work/closure  ← محطة الإغلاق والمتابعة
/dashboard/participation-cases/[id]          ← الملف الكامل بكل التبويبات (admin/إدارة)
```

## أولاً: المحطات (7 صفحات جديدة)

كل محطة = server page + client view خفيف. نمط موحّد:
- استعلام `registrations` مفلتر بـ `case_status` المناسب للمحطة.
- عرض جدول مبسّط (رقم الملف، العميل، الفعالية، الحالة) + زر فتح.
- الفتح ينقل لصفحة محطة الملف: `/dashboard/participation-cases/work/{station}/{id}` تعرض **تبويب واحد فقط** (تبويب المحطة) بدل الـ8.

### قائمة المحطات وفلتر الحالة لكل منها
| المحطة | مسار | تظهر الملفات بحالة | التبويب المعروض |
|---|---|---|---|
| الاستقبال | `/work/intake` | new_request, data_incomplete | (إنشاء + بحث) |
| العملاء | `/work/clients` | data_incomplete, data_complete | tab-client |
| الدفع | `/work/payment` | data_complete, payment_pending, payment_confirmed | tab-payment |
| التسجيل | `/work/registration` | payment_confirmed, registration_in_progress, registration_draft | tab-registration |
| الدعوات | `/work/invitations` | invitation_required, invitation_requested, invitation_received | tab-invitation |
| الفيزا | `/work/visa` | visa_in_progress, appointment_pending, appointment_booked | tab-visa |
| التدقيق | `/work/qc` | final_qc, correction_required, ready_for_next_stage | tab-qc |
| الإغلاق | `/work/closure` | completed, closed, cancelled | (نظرة عامة + سبب الإغلاق) |

### بنية الملفات
```
app/dashboard/participation-cases/work/
  layout.tsx                    ← شريط تبويب علوي يربط المحطات
  intake/page.tsx
  clients/page.tsx
  payment/page.tsx
  registration/page.tsx
  invitations/page.tsx
  visa/page.tsx
  qc/page.tsx
  closure/page.tsx
  components/
    station-list.tsx            ← جدول موحّد قابل لإعادة الاستخدام (يأخذ rows + station)
    station-detail.tsx          ← يعرض تبويب واحد + رأس ملف مبسّط + رابط "الملف الكامل"
```

## ثانياً: مكوّن `station-list.tsx` (مُعاد استخدامه)
- Props: `{ cases, station }`.
- جدول مبسّط: رقم الملف، العميل، الفعالية، حالة الملف. بدون أعمدة الدفع/الموظف (تبقى للملف الكامل).
- زر فتح → `/dashboard/participation-cases/work/{station}/{id}`.
- بحث نصي سريع.

## ثالثاً: مكوّن `station-detail.tsx`
- Props: `{ registration, events, tab }` حيث `tab` هو التبويب المختص للمحطة.
- رأس ملف مبسّط (رقم الملف + اسم العميل + الحالة + زر "الملف الكامل" للإدارة).
- يعرض **تبويب واحد فقط** (يستورد TabClient/TabPayment/... حسب `tab`).
- للمحطات غير المرتبطة بتبويب (intake, closure): عرض مبسّط مخصص.

## رابعاً: صفحات المحطات `work/{station}/page.tsx`
كل واحدة server component:
- فحص الصلاحية (admin يرى الكل، team يحتاج صلاحية المسار).
- استعلام `registrations` مفلتر بـ `case_status IN (...)` المرتبط بالمحطة.
- تمرير لـ `<StationList>`.

## خامساً: صفحات تفاصيل المحطة `work/{station}/[id]/page.tsx`
كل واحدة:
- جلب `registrations` + `registration_events`.
- تمرير لـ `<StationDetail tab="...">`.

## سادساً: الـsidebar والصلاحيات

### `lib/permissions.ts`
إضافة مسارات المحطات إلى `DASHBOARD_PAGES`:
```
{ path: "/dashboard/participation-cases/work/intake", label: "استقبال الطلبات" },
{ path: "/dashboard/participation-cases/work/clients", label: "بيانات العملاء" },
{ path: "/dashboard/participation-cases/work/payment", label: "المالية والدفع" },
{ path: "/dashboard/participation-cases/work/registration", label: "تسجيل الفعالية" },
{ path: "/dashboard/participation-cases/work/invitations", label: "الدعوات" },
{ path: "/dashboard/participation-cases/work/visa", label: "الفيزا والتأمين" },
{ path: "/dashboard/participation-cases/work/qc", label: "التدقيق والجودة" },
{ path: "/dashboard/participation-cases/work/closure", label: "الإغلاق والمتابعة" },
```
بما أن `canAccessPath` يطابق prefix، فصلاحية `/dashboard/participation-cases/work/visa` تمنح الوصول للمحطة + تفاصيلها. الـ admin يرى كل شيء تلقائياً.

### `components/dashboard/sidebar.tsx`
توسيع قسم "إدارة المشاركات":
- **للـ admin**: يعرض "ملفات المشاركة" (الملف الكامل) + كل المحطات.
- **للـ team**: يعرض فقط المحطات المسموحة له في `permissions`.
نستخدم نفس منطقة `canAccessPath` الموجودة، فقط نضيف العناصر للقائمة.

## سابعاً: Layout موحّد للمحطات
`work/layout.tsx`:
- شريط تبويب أفقي يربط المحطات (مثل تبويبات تنقّل).
- يعرض فقط المحطات المسموحة للمستخدم.
- عنوان القسم + وصف موجز.

## ثامناً: التحقق
- `tsc --noEmit`
- `npm run build`
- التأكد أن الـ admin يرى الملف الكامل بالـ8 تبويبات، وأن موظف المحطة يرى محطته فقط.

## ما لا يتغيّر
- مصدر البيانات `registrations` + JSONB — لا تكرار.
- `actions.ts` — نعيد استعمال نفس الإجراءات (searchRegistrations, savePaymentData, saveRegistrationJsonb, ...).
- التبويبات `tabs/tab-*.tsx` — نعيد استعمالها في `StationDetail`.
- صفحة الملف الكامل `[id]/` — تبقى للإدارة بكل التبويبات.
- صفحة القائمة الرئيسية `page.tsx` — تبقى للـ admin.
